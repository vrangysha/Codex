# Codex Prompt Pack: многостраничный лендинг ремонта квартир и домов

Этот файл — отдельный этап работы для Codex. Запускай файлы строго по порядку. Перед началом каждого этапа попроси Codex прочитать уже созданные `AGENTS.md`, `DESIGN.md`, `CONTENT_TODO.md`, `ASSETS.md`, `QA_REPORT.md`, `WORKLOG.md` и текущую структуру проекта.

Общие правила для всех этапов:
- Не делай git commit без отдельного разрешения.
- Не используй платные или закрытые компоненты 21st.dev.
- Не копируй проприетарный код и изображения без лицензии.
- Если данных бизнеса нет, используй аккуратные placeholder-данные и заноси их в `CONTENT_TODO.md`.
- Не выдавай сгенерированные изображения, отзывы и кейсы за реальные.
- После каждого этапа обновляй `WORKLOG.md` и `QA_REPORT.md`.
- Если build/lint/typecheck падают — исправляй, затем запускай повторно.
- Сайт должен быть современным, лаконичным, светлым, доверительным, адаптивным и SEO-готовым.

# 01 — Bootstrap проекта и технический фундамент

## Цель этапа

Подготовь проект к созданию полноценного многостраничного лендинга ремонта квартир и домов. Не начинай дизайн всех страниц сразу. Сначала создай чистый фундамент, структуру, базовые зависимости, правила качества и рабочие документы.

## Задачи

### 1. Изучи проект

Проверь:

- `package.json`;
- структуру папок;
- наличие Next.js, React, TypeScript, Tailwind;
- наличие `app/` или `pages/`;
- наличие shadcn/ui;
- текущий ESLint/Prettier/TypeScript config;
- существующие компоненты и стили;
- текущие routes;
- наличие тестов.

Если проект уже существующий, не переписывай всё с нуля. Сначала зафиксируй найденную структуру в `WORKLOG.md`.

### 2. Если проект пустой

Создай или подготовь стек:

- Next.js App Router;
- TypeScript;
- Tailwind CSS;
- shadcn/ui-compatible структура;
- ESLint;
- базовый форматтер, если его нет;
- директория `src`, если проект так устроен;
- clean app layout.

Рекомендуемая структура:

```text
app/
  layout.tsx
  page.tsx
  services/
  portfolio/
  pricing/
  process/
  about/
  reviews/
  faq/
  contacts/
  privacy/
  terms/
  not-found.tsx
components/
  layout/
  sections/
  ui/
  forms/
  seo/
lib/
data/
public/
  images/
  icons/
```

### 3. Создай рабочие документы

Создай или обнови:

- `WORKLOG.md` — журнал этапов;
- `QA_REPORT.md` — проверки и результаты;
- `CONTENT_TODO.md` — данные, которые должен заменить владелец;
- `ASSETS.md` — источники изображений и статус ассетов;
- `README.md` — запуск проекта;
- `DESIGN.md` — будет заполнен на следующем этапе.

### 4. Инициализируй Impeccable

Если доступен Impeccable:

```text
/impeccable init
```

Затем зафиксируй в `WORKLOG.md`, какие команды доступны. Если Impeccable не установлен или не запускается, запиши это в `QA_REPORT.md` и продолжай вручную.

### 5. Проверь UI UX Pro Max

Если UI UX Pro Max доступен как skill, используй его для следующих этапов дизайна. Если недоступен, в `WORKLOG.md` напиши: “UI UX Pro Max недоступен, применяю принципы вручную”.

### 6. Установи базовые UI-зависимости при необходимости

Если их нет и проект это допускает:

- `lucide-react` для иконок;
- `class-variance-authority`;
- `clsx`;
- `tailwind-merge`;
- `zod` для форм;
- `react-hook-form`, если выбран этот подход;
- shadcn/ui primitives, если они нужны.

Не добавляй тяжёлые зависимости без причины.

### 7. Создай route map

В `WORKLOG.md` опиши будущие страницы:

- `/`;
- `/services`;
- `/services/apartment-renovation`;
- `/services/house-renovation`;
- `/services/cosmetic-renovation`;
- `/services/capital-renovation`;
- `/services/design-project`;
- `/portfolio`;
- `/pricing`;
- `/process`;
- `/about`;
- `/reviews`;
- `/faq`;
- `/contacts`;
- `/privacy`;
- `/terms`.

### 8. Запусти проверки

Выполни доступные команды:

```bash
npm run lint
npm run typecheck
npm run build
```

Если команды отсутствуют, добавь их, если это уместно для проекта.

## Критерии готовности

- проект запускается;
- структура понятна;
- рабочие документы созданы;
- route map описан;
- нет случайных визуальных решений;
- все ограничения записаны;
- `WORKLOG.md` обновлён;
- `QA_REPORT.md` обновлён.
