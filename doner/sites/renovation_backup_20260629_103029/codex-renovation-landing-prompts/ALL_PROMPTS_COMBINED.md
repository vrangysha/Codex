# 00 — Как пользоваться этим набором prompt-файлов

## Назначение

Этот набор нужен, чтобы Codex поэтапно сделал полноценный многостраничный лендинг для бизнеса ремонта квартир и домов: от фундамента проекта и дизайн-системы до отдельных страниц, форм, SEO, изображений, адаптива и финального QA.

## Как запускать

1. Открой проект в Codex.
2. Убедись, что проект чистый или ты понимаешь текущий diff.
3. Вставляй в Codex содержимое файлов по порядку:
   - `01_PROJECT_BOOTSTRAP_PROMPT.md`
   - `02_AGENTS_AND_GLOBAL_RULES_PROMPT.md`
   - `03_DESIGN_SYSTEM_PROMPT.md`
   - и так далее.
4. Не запускай следующий этап, пока предыдущий не завершён и Codex не обновил `WORKLOG.md`.
5. После каждого крупного этапа проси Codex запускать доступные проверки: `lint`, `typecheck`, `build`, `test`.

## Рекомендуемый режим работы

Работай маленькими итерациями:

- этап 1–5: фундамент, дизайн-система, данные, общие компоненты;
- этап 6–18: страницы сайта;
- этап 19–23: юридические страницы, изображения, SEO, формы;
- этап 24–26: адаптив, QA, финальная сдача.

## Если проект пустой

Codex должен создать Next.js + TypeScript + Tailwind CSS проект или использовать существующий шаблон, если он уже есть.

## Если проект уже существует

Codex не должен ломать текущий стек. Он обязан изучить структуру проекта, использовать существующие компоненты и стили, а новые изменения делать аккуратно.

## Подключение скиллов

Перед работой желательно установить:

```bash
npx impeccable skills install
```

После перезапуска Codex:

```text
/impeccable init
```

Для UI UX Pro Max используй установку, указанную в его репозитории. Если skill недоступен в Codex, Codex должен вручную применить его принципы: структура, визуальная иерархия, UX, accessibility, дизайн-система, адаптив, проверка интерфейса.

## Использование 21st.dev

Codex может искать и использовать бесплатные публичные компоненты 21st.dev, особенно для:

- navigation;
- hero;
- feature cards;
- pricing;
- testimonials;
- FAQ;
- forms;
- footer;
- marketing sections.

Правило: любые установленные компоненты нужно привести к единому стилю сайта и проверить.

## Главный критерий

Итог должен быть не набором случайных страниц, а цельным коммерческим сайтом с понятной архитектурой, светлой эстетикой, хорошим UX и проверенным build.

# 02 — AGENTS.md и глобальные правила для Codex

## Цель этапа

Создай проектные инструкции, чтобы Codex во всех следующих этапах работал последовательно, не ломал дизайн и использовал правильные источники компонентов.

## Задачи

### 1. Создай `AGENTS.md`

В корне проекта создай `AGENTS.md` со следующими разделами:

1. Product goal.
2. Target audience.
3. Visual direction.
4. Tech stack rules.
5. 21st.dev usage rules.
6. Impeccable usage rules.
7. UI UX Pro Max usage rules.
8. Accessibility rules.
9. SEO rules.
10. Asset rules.
11. QA rules.
12. “Do not do” list.

### 2. Product goal

Зафиксируй:

- сайт для бизнеса ремонта квартир и домов;
- главная цель — заявки на расчёт стоимости и замер;
- вторичная цель — доверие через портфолио, процесс, гарантии и отзывы;
- язык сайта — русский;
- стиль — светлый, современный, лаконичный, тёплый, премиальный, но без показного люкса.

### 3. Target audience

Опиши аудитории:

- владельцы квартир в новостройках;
- владельцы вторичного жилья;
- семьи, делающие капитальный ремонт;
- владельцы домов;
- клиенты, которым нужен ремонт под ключ;
- клиенты, которые боятся сорванных сроков, смет и некачественных подрядчиков.

### 4. Visual rules

В `AGENTS.md` пропиши:

- светлый интерфейс;
- тёплые нейтральные цвета;
- спокойная сетка;
- качественная типографика;
- CTA заметный, но не кислотный;
- никаких “AI-gradient” фонов;
- никаких случайных neon/glassmorphism решений;
- меньше украшательства, больше доверия.

### 5. 21st.dev rules

Пропиши:

- можно использовать только бесплатные публичные компоненты;
- нельзя использовать закрытые/платные/Pro-компоненты;
- после добавления компонента проверять diff;
- каждый компонент приводить к общей дизайн-системе;
- не оставлять демо-тексты;
- не добавлять тяжёлые зависимости без причины.

### 6. Impeccable rules

Если доступно, использовать:

- `/impeccable shape` перед ключевыми страницами;
- `/impeccable typeset` для типографики;
- `/impeccable layout` для сетки;
- `/impeccable adapt` для адаптива;
- `/impeccable critique` для UX-критики;
- `/impeccable audit` для проверки;
- `/impeccable polish` для финальной полировки.

Если недоступно — выполнять эти проверки вручную.

### 7. UI UX Pro Max rules

Использовать как источник решений для:

- продуктового типа Home Services / Construction / Repair;
- структуры лендинга;
- UX-паттернов лидогенерации;
- цветовой системы;
- типографики;
- accessibility;
- layout density;
- responsive behavior.

### 8. Do not do

Запрети:

- lorem ipsum;
- фейковые “реальные” отзывы;
- фейковые “реальные” кейсы;
- нелицензированные изображения;
- неработающие ссылки;
- декоративные иконки без смысла;
- случайные стили на каждой странице;
- скрытые формы без результата;
- client components без причины;
- незафиксированные TODO.

### 9. Обнови документы

- `WORKLOG.md`: что добавлено;
- `QA_REPORT.md`: какие проверки прошли;
- `CONTENT_TODO.md`: какие реальные данные нужны.

## Критерии готовности

- все будущие этапы имеют единые правила;

# 03 — Дизайн-система лендинга

## Цель этапа

Создай цельную дизайн-систему для сайта ремонта квартир и домов: цвета, типографику, сетку, компоненты, состояния, анимации и правила использования. До создания страниц нужно определить визуальный язык.

## Дизайн-направление

Сайт должен выглядеть как современный сервис ремонта под ключ:

- светлый;
- тёплый;
- лаконичный;
- надёжный;
- визуально дорогой, но не кричащий;
- без шаблонного AI-глянца;
- с ощущением порядка, чистоты и контроля.

## Палитра

Создай CSS variables и Tailwind tokens.

Рекомендуемая палитра:

- `background`: warm white / ivory;
- `surface`: молочный / stone-50;
- `surface-alt`: warm beige / sand;
- `text-primary`: deep graphite;
- `text-secondary`: warm gray;
- `border`: soft stone;
- `accent`: muted terracotta, warm wood или sage;
- `accent-foreground`: ivory;
- `success`: muted green;
- `warning`: muted amber;
- `error`: controlled red.

Проверь контраст WCAG AA.

## Типографика

Выбери шрифтовую пару. Не используй бездумно Inter.

Подходящие варианты:

- `Manrope` для UI;
- `Onest` для UI;
- `Geologica` для modern Russian typography;
- `IBM Plex Sans` для строгого профессионального вида;
- аккуратный serif только для крупных editorial-акцентов, если он не ухудшает читаемость.

Задай:

- H1: крупный, плотный, 48–72 desktop, 36–44 mobile;
- H2: 36–52 desktop, 28–36 mobile;
- H3: 22–28;
- body: 16–18;
- small: 13–14;
- line-height комфортный;
- max-width для текста, чтобы строки не были длинными.

## Сетка и отступы

Определи:

- container max-width: 1120–1280px;
- horizontal padding mobile: 16–20px;
- tablet: 24–32px;
- desktop: 40px;
- section vertical spacing: 72–120px desktop, 48–72px mobile;
- card spacing: 20–32px;
- border radius: 16–28px;
- subtle shadows только там, где нужно.

## Базовые компоненты

Создай или настрой:

- `Button` variants: primary, secondary, ghost, link, outline;
- `Badge`;
- `Card`;
- `SectionHeading`;
- `Container`;
- `LeadButton`;
- `Input`, `Textarea`, `Select`, `Checkbox`;
- `Accordion`;
- `Breadcrumbs`;
- `StatCard`;
- `TrustBadge`;
- `ServiceCard`;
- `ProjectCard`;
- `PriceCard`.

## Состояния

Для всех интерактивных элементов:

- hover;
- focus-visible;
- active;
- disabled;
- loading;
- error;
- success.

Focus должен быть видимым и аккуратным.

## Анимации

Используй минимально:

- мягкий fade-up для секций;
- hover lift для карточек;
- плавные accordion transitions;
- уважай `prefers-reduced-motion`.

Запрещено:

- bounce;
- flashy parallax;
- постоянное движение;
- тяжёлые animation libraries без необходимости.

## Документация

Заполни `DESIGN.md`:

- принципы;
- палитра;
- типографика;
- spacing;
- компоненты;
- состояния;
- анимации;
- accessibility;
- примеры использования.

## Использование Impeccable/UI UX Pro Max

Если доступны:

```

```

- `/impeccable init`
- `/impeccable shape`
- `/impeccable typeset`
- `/impeccable layout`
- `/impeccable adapt`
- `/impeccable critique`
- `/impeccable audit`
- `/impeccable polish`
- `/impeccable optimize`

If unavailable, apply the same process manually and document it.

## UI UX Pro Max rules

Use the skill, if available, for product type, UI style, UX patterns, palette, typography, accessibility, and responsive checks. If unavailable, manually apply professional UI/UX reasoning.

## Content rules

- Language: Russian.
- Tone: calm, professional, specific, trustworthy.
- No Lorem Ipsum.
- No fake real reviews or fake real projects.
- Placeholder content must be documented in `CONTENT_TODO.md`.

## Asset rules

- Use licensed public images or generated images.
- Record every source in `ASSETS.md`.
- Add alt text.
- Optimize images.
- Do not present generated portfolio as real work.

## QA rules

Before final delivery:

- lint;
- typecheck;
- build;
- responsive check;
- accessibility check;
- metadata check;
- links check;
- form validation check;
- image/license check;
- no secrets check.

Update `QA_REPORT.md` after every major stage.
