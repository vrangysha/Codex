# Codex Prompt Pack: многостраничный лендинг ремонта квартир и домов

Этот файл — отдельный этап работы для Codex. Запускай файлы строго по порядку. Перед началом каждого этапа попроси Codex прочитать уже созданные `AGENTS.md`, `DESIGN.md`, `CONTENT_TODO.md`, `ASSETS.md`, `QA_REPORT.md`, `WORKLOG.md` и текущую структуру проекта.

Общие правила для всех этапов:
- Не делай git commit без отдельного разрешения.
- Не используй платные или закрытые компоненты 21st.dev.
- Не копируй проприетарный код и изображения без лицензии.
- Если данных бизнеса нет, используй аккуратные placeholder-данные и заноси их в `CONTENT_TODO.md`.
- Не выдавай сгенерированные изображения, отзывы и кейсы за реальные.
- После каждого этапа обновляй `WORKLOG.md` и `QA_REPORT.md`.
- Если build/lint/typecheck падают — исправляй, затем запускай повторно.
- Сайт должен быть современным, лаконичным, светлым, доверительным, адаптивным и SEO-готовым.

# 26 — Финальный отчёт Codex после завершения лендинга

## Цель этапа

Сформировать финальный отчёт для владельца проекта: что сделано, где лежит, что нужно заменить, какие проверки пройдены, какие ограничения остались.

## Создай файл `FINAL_REPORT.md`

Структура:

### 1. Кратко

- что создано;
- стек;
- количество страниц;
- статус build;
- статус QA.

### 2. Созданные страницы

Таблица:

| Страница | Route | Статус | Комментарий |
|---|---|---|---|

Включи все routes.

### 3. Компоненты

Список:

- layout;
- header;
- footer;
- hero;
- services;
- portfolio;
- pricing;
- process;
- FAQ;
- forms;
- CTA;
- SEO components.

### 4. 21st.dev

Если использовались компоненты:

- название;
- URL/registry;
- где используется;
- что изменено;
- лицензия/условие, если известно.

Если не использовались:

- написать почему: “подходящие бесплатные компоненты не использованы / создано вручную для консистентности”.

### 5. Impeccable

Указать команды, если использовались:

- init;
- shape;
- typeset;
- layout;
- adapt;
- critique;
- audit;
- polish;
- optimize.

Если недоступно — указать, что аналогичный чеклист выполнен вручную.

### 6. UI UX Pro Max

Указать, как применён:

- product type;
- style;
- palette;
- typography;
- UX patterns;
- accessibility;
- responsive.

Если skill недоступен — указать manual application.

### 7. Проверки

Таблица:

| Проверка | Команда | Результат | Комментарий |
|---|---|---|---|

Включи:

- lint;
- typecheck;
- test;
- build;
- responsive;
- accessibility;
- SEO;
- links;
- forms;
- assets;
- no secrets.

### 8. Что заменить владельцу

Список из `CONTENT_TODO.md`:

- название компании;
- город;
- телефон;
- email;
- адрес;
- юридические данные;
- цены;
- реальные фото;
- реальные кейсы;
- реальные отзывы;
- политика конфиденциальности;
- CRM/webhook;
- analytics;
- домен/base URL.

### 9. Ограничения

Указать честно:

- какие данные placeholder;
- какие изображения generated/placeholder;
- что требует юридической проверки;
- что требует подключения backend/CRM;
- что требует реальных кейсов.

### 10. Следующие шаги

- заменить контент;
- подключить домен;
- подключить аналитику;
- подключить CRM/Telegram/email;
- добавить реальные кейсы;
- прогнать Lighthouse на production build;
- проверить юридические тексты.

## Финальный ответ в Codex

После создания `FINAL_REPORT.md` дай краткий ответ:

1. Готово.
2. Что создано.
3. Какие проверки прошли.
4. Где смотреть отчёты.
5. Что заменить перед публикацией.

## Критерии готовности

- `FINAL_REPORT.md` создан;
- все TODO прозрачны;
- пользователь понимает, как запускать и публиковать сайт;
- никаких скрытых ограничений.
